\chapter{PROJECT RESULTS AND EVALUATION}

This chapter presents the results of the Autonomous Parcel Delivery System, developed and tested across five progressive phases. Each phase built directly upon the results of the previous one, starting from a simple Python script and ending in a full robotics simulation. The goal throughout was to ensure that every component worked correctly before adding the next layer of complexity.

\section{Phase 1: Path Planning in VS Code}

In the first phase the path planning logic was written and implemented in a Python environment using Visual Studio Code. This was intentionally done by keeping things simple at first, any bugs in the algorithm could be caught and fixed without the added complexity of ROS 2 or hardware.Everything was done in parts and then integrated together The planning process was broken into four steps.

% ─── Step 1 ──────────────────────────────────────────────────────────────────
\subsection{Step 1: Exploring the Map with A*}

The first task was to find a route through the map. The Improved A* (IA*) algorithm was used for this purpose. It works by starting at the robot's position and expanding outward, checking which directions lead closer to the goal while avoiding obstacles.

The red crosses in the figure \ref{fig:astar} represent every grid cell the algorithm checked before finding the solution. The best thing to observe is the manner in which the search went around the barrier in the middle of the map.The fact that the A-star managed to navigate around the obstacle is a confirmation that the heuristic functionality is properly functioning and leading the search in a reasonable direction.

	\caption{A* node expansion showing exploration process}
	\label{fig:astar}
\end{figure}

The figure shows every cell visited during the search process, marked with red crosses. The algorithm successfully avoids the U-shaped trap in the center and reaches the goal, indicated by the blue star. This demonstrates the effectiveness of the heuristic function in guiding the search toward the goal while navigating around complex obstacles.

Once the goal was found, the algorithm traced back through its steps to draw the route it had taken. This is called the raw path, shown as a green line in the figure.

corners would make it come to a halt and turn and then come to a halt again each turn. So although
this is the step that the planner was sure that he has walked the correct path.

% ─── Step 3 ──────────────────────────────────────────────────────────────────
\subsection{Step 3: Cleaning Up the Path (Pruning)}

The next step was to simplify the path by removing unnecessary waypoints. The pruning algorithm goes through the list of points and checks whether it is possible to go directly from one point to another further ahead without passing through an obstacle. If yes, all the points in between are removed since they are not needed.

The result is a blue dashed line with far fewer points, and the long unnecessary zigzags are gone. This makes the path much easier for the robot to follow and reduces the amount of computation needed later.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.7\textwidth]{pruned.png}
	\caption{Pruned path with redundant waypoints removed}
	\label{fig:pruned}
\end{figure}

The blue dashed line shows the result of the pruning process. Redundant waypoints have been removed, giving a cleaner and shorter route. This simplified path is far easier for the motion controller to process and follow, significantly reducing computational overhead while maintaining obstacle avoidance.

% ─── Step 4 ──────────────────────────────────────────────────────────────────
\subsection{Step 4: Smoothing the Path}

Despite the pruning methods employed, the final composition remained a concatenation of rectilinear stretches which ended with angular corners. The final step involved the skillful integration of each of these parts into a one continuous line by use of cubic spline interpolation.

The result is the blue curved trajectory shown in the figure. Instead of sharp corners, the robot now has a gentle, arc-like path to follow. This is important because a robot moving along a smooth curve does not need to slow down at every turn, it can maintain a steady speed throughout the journey. This smoothed trajectory is what gets handed to the next phase.

\section{Phase 2: Motion Control with MPC}

With a clean reference path in hand, the next challenge was making sure the robot could actually follow it accurately so for that a Model Predictive Controller (MPC) was implemented for this purpose.

The key difference between MPC and a simpler controller like PID is that MPC looks ahead. Instead of only reacting to where the robot currently is, it uses a short-term prediction of where the robot will be over the next few steps and adjusts its movements accordingly. This makes it much better at handling curves, because it can start slowing down or adjusting heading before a turn, not after it has already missed it.

\subsection{Keeping the Robot Safe: Obstacle Inflation}
The obstacle inflation is an essential safety measure that is incorporated into the navigation backend. Instead of the physical barrier of an obstruction as a boundary the system is run over a mathematical expansion of the footprint of the obstacle thus establishing a virtual no-go buffer zone.

Figure \ref{fig:inflation} visualizes this mechanism clearly. The diagram differentiates between two distinct zones:
\begin{itemize}
	\item \textbf{Original Obstacle:} 
	The core, which was marked by a light pink colour, is used to outline the real physical size of the object.
	
	\item \textbf{Inflated Obstacle:} 
	The outer representation, which is shown in deeper red, represents the safety buffer that the algorithm introduces.
\end{itemize}

By the hard limit is defined as the Inflated edge, the path planner is pushed to maintain a safe distance of the physical object of the robot. This essentially supplies the uncertainties encountered in the real world, such as sensor noise, or small localization errors, and thus the robot (initially in the green marker) will never hit the wall when working towards the goal (red square).

\section{2D Global Planner Results}

We ran the global planner on the 2D map environment first before entering the full 3D simulation. The aim of those tests was ensuring that the planner is capable of producing safe and correct paths in varying obstacle layouts. A 2D version allows us to exercise sanity of the path planning code without wasting time on robot dynamics, sensor noise or real-time control issues.

In order to assess the planner correctly we established three scenarios. Each scenario was tailored to test a unique factor, one that entailed obstacle avoidance with an adequate safety margin, one to avoid blocking obstacles, and the final was to plan distances on a larger scale on a cluttered map.
\subsection{Cellular Decomposition Output (2D Workspace Partitioning)}
In order to make global navigation in a complex indoor setting simple, we adopted a cellular decomposition method on the 2D workspace. The primary aim of this step is to divides into several non-overlapping areas (cells) the free space according to the boundaries of obstacles. The cells are safe zones where the robot will not run into anything.

We then pick a representative point (cell center) of each region after decomposition. Adjacency relations are then used to construct a connectivity graph out of these centers.This graph provides a simplified high-level structure of the environment and helps the
planner efficiently compute a valid route from the start position to the goal.

Figure~\ref{fig:cellular_decomposition} shows the final output of the cellular decomposition
process, including the generated cells, their connectivity, and the resulting global path.
The planned route demonstrates how the robot transitions between cells while avoiding
obstacles and restricted regions.

\section{Phase 5: ROS 2 and Gazebo Simulation Results}

The final phase moved all previously validated planning and control algorithms into a fully integrated ROS~2 and Gazebo simulation. This milestone marks the shift from isolated two-dimensional testing to a complete stack, combining SLAM, simulated LiDAR and camera sensing, and our own IA*-based global planning, pruning, spline smoothing, and MPC control pipeline, implemented as custom ROS~2 nodes rather than relying on any off-the-shelf navigation planners, to autonomously carry out parcel delivery missions inside a custom multi-room indoor world.

\subsection{Custom Multi-Room Delivery Environment}

A custom indoor world was built in Gazebo to emulate a realistic multi-room facility such as an office floor. The environment consists of five distinct rooms, colour-coded red, green, gray, blue, and teal, connected by a shared central corridor and enclosed by an outer boundary wall. \section{Challenges Faced}

Throughout the development of the autonomous parcel delivery system, several technical challenges were encountered during the implementation and integration of different software components. These challenges required extensive debugging, parameter tuning, and iterative testing before a stable and reliable system was achieved.\begin{itemize}
\item \textbf{Obstacle Collision During Path Planning:}
During the early stages of path planning, some planned paths passed too close to obstacles, increasing the risk of collision. This issue was resolved by implementing obstacle inflation and defining suitable safety margins, allowing the robot to maintain a safe distance while navigating to the goal.

\item \textbf{Path Smoothing Issues:}
Although the Improved A* algorithm generated valid collision-free paths, the resulting trajectories contained sharp turns that were difficult for the robot to follow smoothly. Cubic spline interpolation was used to generate smoother paths while maintaining a safe distance from obstacles. Several iterations were required to achieve a balance between smoothness and path accuracy.

\item \textbf{Gazebo Installation and Simulation Setup:}
Setting up the Gazebo simulation environment was one of the initial challenges of the project. The simulator occasionally displayed a black screen or failed to load the world due to rendering and resource configuration issues. These problems were resolved by selecting the correct rendering engine, configuring resource paths, and installing the required dependencies.

\item \textbf{Custom World Development:}
Creating a realistic indoor environment required designing rooms, corridors, furniture, doors, and moving human actors while ensuring enough space for the robot to navigate safely. Several adjustments were made to model placement, object sizes, and collision properties before a stable simulation environment was achieved.

\item \textbf{Motion Controller (MPC) Tuning:}
Although the global planner generated suitable paths, the robot initially struggled to follow them accurately. The MPC controller required careful tuning of prediction horizons, control weights, and velocity limits before the robot could smoothly track the planned path in different environments.

\item \textbf{ROS~2 and Web Application Integration:}
Connecting the Flutter web application with the ROS~2 navigation system was another major challenge. Reliable communication between the Flutter frontend, Flask backend, and ROS~2 nodes was necessary to send delivery requests, receive robot status updates, and display live information correctly.


\item \textbf{Package Dependency Conflicts:}
The project used several ROS~2 packages and third-party libraries, which occasionally caused dependency conflicts during installation and updates. These issues were resolved by installing compatible package versions and rebuilding the workspace whenever required.

\item \textbf{Launch File Configuration:}
Managing multiple launch files for Gazebo, SLAM Toolbox, Nav2, the Flask backend, and custom ROS~2 nodes increased the complexity of the system. Careful organization and repeated testing were required to ensure that all components started in the correct order and worked together properly.

\item \textbf{System Integration and Testing:}
The final challenge was integrating all system components, including path planning, motion control, SLAM, Gazebo, ROS~2, the Flask backend, and the Flutter web application, into a single working system. Extensive testing was carried out to verify that all modules communicated correctly and that the robot could successfully complete delivery missions from start to finish.

\end{itemize}

Despite these challenges, continuous debugging, parameter tuning, and repeated testing resulted in a fully integrated autonomous parcel delivery system capable of reliable navigation, obstacle avoidance, and real-time user interaction through the web application.
